# 项目优化循环流程

> 适用阶段：solo founder / 早期产品（v0.x）/ 多项目并行
> 核心原则：**每次对话即一次优化机会，每周一次跨项目综合，每版一次深度审计**

---

## 为什么不用纯"按天"或纯"按版本"

| 方式 | 优点 | 对当前阶段的问题 |
|------|------|-----------------|
| 按天 | 节奏固定、反馈快 | solo founder 一天可能跨 3 个项目，也可能一天只做 1 个；强行按天会产生空转或遗漏 |
| 按版本 | 单元完整、有交付感 | 早期版本边界模糊，一个"功能"可能跨 3 天，中间累积的优化机会被延迟 |

**结论：用"会话（session）+ 周关卡（weekly gate）+ 版本里程碑（version milestone）"三层节奏。**

---

## 三层优化节奏

### 第一层：会话级优化（每次 vibe coding 结束后触发）

**触发条件**：你和一个项目完成了一轮 vibe coding 对话，准备切到下一个项目或结束工作。

```
对话结束
  ↓
① 回顾本次对话，识别 3 类优化点
  ↓
② 分类处理
  ↓
③ 归档 → 进入下一轮
```

#### ① 识别优化点（3 类）

| 类型 | 说明 | 示例 |
|------|------|------|
| **代码质量** | 对话中发现的坏味道、重复、隐患 | "这段错误处理太粗糙" |
| **流程/规则** | 对话中暴露的规则缺失或过时 | "CLAUDE.md 没覆盖这种场景" |
| **需求/产品** | 对话中发现的用户需求理解偏差 | "这个功能的真实场景和假设不同" |

#### ② 分类处理

| 优化类型 | 处理方式 | 是否需要 PR |
|----------|----------|------------|
| 代码质量（小修） | 直接在同一会话中修复 | 不需要，随本次改动提交 |
| 代码质量（重构级） | 记录到项目 OPTIMIZE.md | 需要，单独 PR |
| 流程/规则 | 更新项目 CLAUDE.md 或规则文件 | 需要，规则 PR |
| 需求/产品 | 记录到项目 BACKLOG.md | 暂不需要，等下一轮规划 |

#### ③ 会话级输出物

每个项目维护一个 `OPTIMIZE.md`（优化待办清单）：

```markdown
# OPTIMIZE.md — {项目名}

## 待优化（未开始）

- [ ] [P1] 重构 collector daemon 的 FIFO 错误处理（来源：2026-05-19 对话）
- [ ] [P2] dashboard API 缺少分页（来源：2026-05-19 对话）

## 进行中

- [ ] [P0] ...

## 已完成

- [x] [P1] 添加 Codex 同步 — PR #12 — 2026-05-18
```

### 第二层：周关卡优化（每周五，固定 1 小时）

**触发条件**：每周五下午，或你本周最后一次工作会话前。

**目的**：跨项目俯瞰，处理积累的优化项，防止优化债堆积。

```
周五回顾
  ↓
① 各项目 OPTIMIZE.md 扫描
  ↓
② 跨项目模式识别
  ↓
③ 批量处理 / 批量生成 PR
  ↓
④ 更新 Rule Feedback
```

#### 周关卡检查清单

- [ ] 扫描每个项目的 `OPTIMIZE.md`，统计 P0/P1 待办数
- [ ] 识别跨项目的共性问题（如"所有项目都缺测试"）
- [ ] 处理 Sidecar 生成的 rule feedback（`/api/rule-feedback`）
- [ ] 合并本周已批准的 PR，验证通过
- [ ] 将 BACKLOG 中成熟的条目提升到下一周的开发计划
- [ ] 更新 Sidecar 的 `CLAUDE.md` 规则（接受有效的 rule feedback）
- [ ] 写一条简短的周记（1-3 句）到 `WEEKLY.md`

#### 优先级排序规则

```
P0 堵塞问题（必须本周处理） > P1 跨项目共性问题 > P1 单项目优化 > P2 所有其他
```

#### PR 批量策略

- 同一项目的多个小优化 → 合并为 1 个 PR（标题：`chore: weekly optimization batch`）
- 跨项目的同类修改（如更新所有项目的 CLAUDE.md）→ 逐项目各 1 个 PR
- 大型重构 → 单独 PR，标注 `refactor:` 前缀

---

### 第三层：版本里程碑优化（每个版本发布前触发）

**触发条件**：项目即将发布新版本（哪怕是 v0.1 → v0.2）。

**目的**：发布前的质量关卡，确保每个版本都比上一个版本更好。

```
准备发布
  ↓
① 运行完整优化审计
  ↓
② 处理所有遗留优化项
  ↓
③ 更新版本号 + changelog
  ↓
④ 发布 + 部署验证
```

#### 版本里程碑检查清单

- [ ] `OPTIMIZE.md` 中无 P0 未处理项
- [ ] `BACKLOG.md` 中无被遗忘超过 2 周的条目（要么做，要么删）
- [ ] 运行 Sidecar 的 Startup Audit，确认 score ≥ 70
- [ ] 运行测试套件，确认全部通过
- [ ] 更新 `CHANGELOG.md`
- [ ] 更新版本号
- [ ] 合并所有待合并 PR
- [ ] 部署并验证（canary check）

```
│  Session 级 ──→ 每次对话结束 → 识别 → 小修/记录     │
│      │                                             │

```

## 与 Sidecar 的集成

Sidecar 的 Rule Feedback 机制天然适配这个循环：

| 优化循环层级 | Sidecar 提供的信号 | 你的动作 |
|-------------|-------------------|---------|
| Session 级 | 对话中的 role quality 评分 | 即时微调对话方式 |
| Weekly Gate | Weekly CEO Review、rule feedback 列表 | 审阅并 apply 有效的 rule feedback |
| Version Milestone | Startup Audit、Organization Audit | 全面评估，更新规则和流程 |

## Solo Founder 实操节奏（建议）

假设你管理 N 个项目：

### 日常工作流

```
周一～周四：每个项目进行 vibe coding 对话
  - 每完成一个项目的对话 → 花 5 分钟更新该项目的 OPTIMIZE.md
  - 小修直接在对话中完成
  - 大优化记录下来，等周五批量处理
```

### 周五流程（~1 小时）

```
1. [15 min] 扫描所有项目的 OPTIMIZE.md 和 Sidecar dashboard
2. [15 min] 识别跨项目模式，排序优先级
3. [20 min] 批量生成 PR 或直接修复
4. [10 min] 写周记，更新规则
```

### 发版流程（不定期，按需）

```
1. 确认 OPTIMIZE.md 清零（至少 P0/P1 清零）
2. 运行完整审计
3. 更新版本号和 changelog
4. 发布
```

---

## 启动检查

在开始使用这个流程前，确认以下就绪：

- [ ] 每个项目都有 `OPTIMIZE.md`（没有就创建一个空的）
- [ ] 每个项目都有 `BACKLOG.md`（用于存放需求级优化）
- [ ] Sidecar dashboard 可访问（`http://localhost:4041`）
- [ ] Sidecar hooks 已安装（`node bin/install-claude-hooks.ts`）
- [ ] 你有一个固定的时间做 weekly gate（建议周五下午）
