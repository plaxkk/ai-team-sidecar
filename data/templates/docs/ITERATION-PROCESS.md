# 5 角色迭代流程

## 标准流程（5 个角色）

每个功能需求必须经过以下 5 个角色的完整流程：

```
Product → Engineer → Creative Review → QA → Tech Lead
```

## 各角色产出标准

### 1. Product（产品理解）
- Engineering Task Spec（7 个必填字段）
- 优先级标注（P0/P1/P2）
- 至少 1 个不确定点或假设

### 2. Engineer（技术方案）
- 系统理解说明
- 至少 2 个技术方案 + 推荐
- 代码修改（diff 或代码示例）
- Spec 覆盖确认

### 3. Creative Review（创意审查）
- 至少 2 个备选方案对比
- 反面意见 / 魔鬼代言人分析
- 用户证据引用
- 商业价值评估
- 可行性评估

### 4. QA（质量审查）
- 至少 3 个检查项（标注风险等级）
- 至少 1 个潜在问题
- 至少 2 个边界情况

### 5. Tech Lead（决策收口）
- 方案总结（引用各角色结论）
- QA 风险摘要
- 明确的 Go/No-Go 建议
- 需要用户确认的决策点

## 简化流程（3 个角色）

适用于 bugfix、部署等任务：

```
Engineer → QA → Tech Lead
```

## 最小流程（1 个角色）

适用于继续、微小调整等：

```
Tech Lead（总结状态 + 下一步动作）
```

## 迭代原则

1. **每个 episode 只对应一个可验收交付物**
2. **交付结尾固定包含：验证证据 + 剩余风险 + go/no-go**
3. **返工超过 2 次时暂停，先总结决策再继续**
4. **每周使用 Weekly CEO Review 复盘**
